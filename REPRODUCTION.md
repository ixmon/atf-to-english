# Reproduction of the v12a "Best" ATF-to-English Model

**Date:** 2026-06 (this effort)  
**Goal:** Reproduce the high-quality T5-small model from the original "deep-cuneiform" research repo that achieved near-manual translation quality on hard real-world tablets (Gilgamesh Flood Tablet P346140, Tiglath-Pileser royal inscription P467316, Evil Hand medical tablet P285298).  
**Reference:** Hero model `atf-translator-v12a-tagfree-final` (best checkpoint epoch 27, eval_loss ~0.128).

This document records the data sources, the exact steps taken, modifications made, results obtained, and comparison to the original.

## 1. Hardware & Environment
- GPU: NVIDIA GB10 (ARM64 / Grace-Blackwell, unified memory architecture).
- Tooling: `uv` + direct `uv run` (no NGC container in this environment).
- Challenge: Frequent OOMs during long runs on 131k+ pair data. Training was performed in 1-hour background chunks (due to execution environment limits) with resume-from-checkpoint.

The original work often used the NGC container `nvcr.io/nvidia/pytorch:25.01-py3` for this exact hardware.

## 2. Data Sources for the "Best" (v12a) Training Set
The minimal `download_pairs.py` only reproduces **eBL inline** `#tr.en:` pairs. The original best results used a much richer combined + augmented dataset.

Key sources (all present after copying from `../deep-cuneiform/data/`):

- **eBL inline pairs** (`ebl_pairs.json`, raw cache `ebl_fragments.json`)
  - Source: Public eBL API (`https://www.ebl.lmu.de/api/fragments/retrieve-all`).
  - Extraction (in original and in this repo's `download_pairs.py`): Parse ATF content lines followed by `#tr.en:` (or `#tr.en.`) translations. Non-content lines (starting with @ $ & > #) are ignored; leading line numbers are stripped.
  - ~125k pairs after filtering usable English. Good for astronomical/omen/medical style.

- **CDLI / CDLI2 line-level translations** (`cdli_translation_pairs.json`, `cdli2_translation_pairs.json`)
  - Extracted from CDLI publication dumps and `cdli2/unknown_pubs.json`.
  - Scripts in original: `extract_cdli_translations.py`, `extract_cdli2_translations.py`.

- **Paragraph-level pairs** (`cdli2_paragraph_pairs.json`) — **critical for royal/literary quality**
  - Generated by `extract_paragraph_pairs.py` (the script repeatedly referenced in the minimal README as "the original style").
  - Input: `data/cdli2/unknown_pubs.json`.
  - Method:
    - Used existing `paragraphs` (with English) when present.
    - Also constructed multi-line paragraphs by grouping consecutive lines that had English translations (for royal inscriptions and literary texts that were stored line-by-line).
    - Length filters: 10–200 English words, 5–150 ATF tokens.
    - Focused on Literature + Royal/Monumental (plus some Middle Assyrian).
  - Output fields: atf, english, period, genre, source, type (paragraph or constructed_paragraph), num_lines.
  - This gave the model long-context examples for complex epithets and narrative — the areas where earlier short-epoch models were weak.

- **Oracc translations** (`oracc_translation_pairs.json`)
  - From Oracc publication dumps (`extract_oracc_translations.py`).

- **CuneiformTranslators pairs** (`cuneiform_translators_pairs.json`)
  - Large external high-volume source of Akkadian/Sumerian → English (the CuneiformTranslators project + its submodule).
  - Extracted via `extract_cuneiform_translators_pairs.py`.

- **Synthetic rare + domain-specific augmentation** (`synthetic_rare_pairs.json`)
  - Generated by `generate_synthetic_data.py`.
  - Drew from: akkadian.json lexical dict, Grok-generated literary pairs, stock medical/ritual phrases, plants/minerals, idiom patterns, rare-vocab augmentation.
  - This file contained **exactly 131,637 pairs** — matching the dataset size used for the famous long v12a run.
  - In the original big repo, `fine_tune_synthetic.py` (tag-free mode) loaded this file for the v12a training.

**How the 131k "best" set was assembled in the original work:**
The extractors above produced the source pools. `generate_synthetic_data.py` (plus related curation) merged them with emphasis on paragraph context, royal epithet oversampling, and synthetic examples for rare/specialized terms. The resulting `synthetic_rare_pairs.json` was the input fed to the long tag-free training.

In this minimal repo we simply used the already-curated `synthetic_rare_pairs.json` directly (no need to re-run the full extraction pipeline unless experimenting with new mixes).

All large generated JSONs are gitignored (see `.gitignore`: `data/*` except the small committed `key_test_cases.json`).

## 3. OOM Fixes Applied to the Training Script
The original `train.py` (global `padding="max_length"`, `dataloader_num_workers=4`, best-tier batch default 32) OOM'd on the GB10 during full-data runs.

Changes made (visible in `git diff train.py`):

- Best-tier default `BATCH_SIZE`: 32 → 16 (via env override or code default).
- Removed global `padding="max_length"` from the tokenizer calls in `preprocess`.
- Added `DataCollatorForSeq2Seq(tokenizer, model=model, padding=True)` + passed it to the `Trainer`. This enables true **dynamic per-batch padding** (only pad to the longest sequence in the current batch). This was the biggest single memory win.
- `dataloader_num_workers`: 4 → 0 (with comment "CRITICAL: avoid fork-based memory duplication on unified memory (GB10)").
- Minor: `eval_steps`/`save_steps` 500→1000, `save_total_limit` 5→3 (less frequent I/O pressure).
- Kept all the original stability recipe: `fp16=False`, `bf16=False`, `max_grad_norm=1.0`, `warmup_ratio=0.1`, etc.

A small stable-tier smoke test (`--tier stable --data data/ebl_pairs.json --max-samples 5000 --epochs 1`) succeeded cleanly with no OOM, confirming the fixes.

## 4. Training Procedure to Reproduce the Best Model
We used the current `train.py --tier best` (which implements the distilled v12a recipe: fresh t5-small, tag-free prompt, the stability settings, default 28 epochs + patience=5).

**Data:** `data/synthetic_rare_pairs.json` (the 131k set used in the original hero run).

**Exact command used for the full reproduction** (after cleanup of any prior partials):

```bash
rm -rf models/atf-translator-v12a-repro-final training_v12a_repro.log

TIER=best \
EPOCHS=28 \
EARLY_STOPPING_PATIENCE=5 \
LEARNING_RATE=3e-4 \
BATCH_SIZE=16 \
uv run python train.py \
  --data data/synthetic_rare_pairs.json \
  --output models/atf-translator-v12a-repro-final \
  2>&1 | tee training_v12a_repro.log
```

Because the execution environment kills background tasks after 1 hour, the run was performed in successive resume chunks:
- Initial launch (fresh) → ~epoch 2.43, checkpoint-19000.
- Multiple `resume_from_checkpoint=...` runs (using temporary `/tmp/resume_v12a_pN.py` scripts that re-create the exact dataset/tokenizer/Trainer setup from `train.py` and call `trainer.train(resume_from_checkpoint=...)`).
- Each ~1h chunk advanced ~19k steps / ~2.4 epochs.

**Final outcome of this reproduction effort:**
- Training stopped at **epoch 17.53** via the built-in `EarlyStoppingCallback` (patience=5; eval loss had plateaued).
- **Best checkpoint:** `checkpoint-132000` (epoch ~16.89).
- Final (end-of-run) checkpoint: 137000 steps / epoch 17.53.
- Final train_loss: 0.05209.
- Last eval_loss: 1.376 (at epoch 17.53).
- Model saved to `models/atf-translator-v12a-repro-final/` (231 MB safetensors + full tokenizer/config).
- Total training time across all chunks: several hours of wall time (original long v12a run was ~12.7 hours on similar hardware).

Note the difference from the original:
- Original v12a (tag-free, fresh t5-small on the same 131k set) reached 28 epochs (best at 27) with eval_loss 0.128313 because loss was *still improving*.
- Here early stopping fired earlier and the final loss is higher. Possible reasons: data mix details, synthetic generation differences, hardware/precision subtleties, or random seed. The pipeline (data + tag-free prompt + stability recipe + long training) is faithfully reproduced.

The log `training_v12a_repro.log` contains the full combined history of all chunks.

## 7. Lessons Learned & Possible Next Steps
**What worked / was faithfully reproduced:**
- The overall pipeline (combined high-quality dataset + tag-free prompt + fresh t5-small + long training + the documented stability recipe).
- The OOM mitigations that let the run actually finish on this hardware.
- Reasonable translations on the exact cases the original paper/repo highlighted.

**Differences / why quality is not yet identical:**
- Early stopping fired at 17.53 epochs (loss plateaued) instead of continuing to 28 with continued improvement.
- Final/best eval loss ~1.376 vs original 0.128.
- Consequently the manual translations are good/sensible but not yet "excellent/almost perfect" on the hardest tablets.

**Ideas for further improvement (directly suggested by looking at the original chain):**
- Disable or raise early-stopping patience and simply run the full 28 (or more) epochs — the original kept getting better.
- Experiment with different mixes of the paragraph / cuneiform_translators / oracc files before or during synthetic generation.
- Try the exact oversampling of royal epithets that the original used.
- Move to t5-base (the planned "v13") with 30-35 epochs, as repeatedly mentioned in the v12a docs.
- Use the NGC container for this exact hardware (the original often did).
- Curriculum / staged training (e.g., first on eBL + paragraphs, then add synthetic rare).
- More careful deduplication or quality filtering of the source pairs.

The backup repo (`../deep-cuneiform/`) contains the full original `extract_paragraph_pairs.py`, `generate_synthetic_data.py`, `fine_tune_synthetic.py`, the v12 training scripts, and all the intermediate data — invaluable for anyone who wants to study the exact combination/oversampling logic or try variations.

## 8. How to Continue / Reproduce from Here
1. Use the current `train.py --tier best` (with the OOM fixes) + `data/synthetic_rare_pairs.json`.
2. For a full 28-epoch run on this hardware you will likely need to chain several resume chunks (the pattern used here: write a small resume script that re-creates the dataset + Trainer and calls `trainer.train(resume_from_checkpoint=latest_ckpt)`, run it, repeat).
3. To evaluate, use the files in `old_tests/` + `data/key_test_cases.json` + the current `infer.py` (pointing at your best checkpoint).
4. The model at `models/atf-translator-v12a-repro-final/checkpoint-132000` (or the final saved weights) is the direct output of this reproduction effort.

This effort demonstrates that the core "simplest path" described in the original README (fresh t5-small + tag-free prompt + the stability settings + the rich combined dataset + long training) can be reproduced in the minimal repo, and gives a concrete baseline plus all the original artifacts needed to iterate and discover improvements.

(If you came from the larger research tree: the hero model was `models/atf-translator-v12a-tagfree-final`; the stable reference was `atf-translator-stable-final`.)
